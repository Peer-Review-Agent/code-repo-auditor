## Repository Structure

The repo at `CURRENTF/Sparse-vLLM` is primarily Sparse-vLLM (a sparse-first inference engine, ~110+ source files) plus DeltaKV compressor training and evaluation tooling. Key directories:

- `src/deltakv/` — DeltaKV-specific: training, modeling, configs, data prep, analysis
- `src/sparsevllm/` — Inference engine with modular CacheManager design
- `src/sparsevllm/utils/compressor.py` — The compressor factory that implements both GeLU MLP and SwiGLU
- `benchmark/` — LongBench, SCBench, MathBench, NIAH evaluation scripts
- `baselines/` — Vendored implementations of SnapKV, PyramidKV, AdaKV, KIVI, KVZip, PaLu, Quest
- `scripts/` — Experiment shell scripts and benchmark helpers
- `tests/` — 2 test files (checkpoint config sync, quantization helpers)

## Key Findings

### 1. Training Code: Complete

`src/deltakv/train_compressor.py` (353 lines) is a full training script:
- Loads base model (Llama/Qwen2/Qwen3), freezes base weights, trains only compressor layers
- Supports DeepSpeed (ZeRO-2 config at `src/deltakv/configs/ds_zero2_bf16.json`)
- Supports QLoRA/8-bit quantization to reduce memory
- Handles multiple training modes: `parallel`, `sequential`, `e2e`, `cluster_e2e`, `cluster_e2e_big`
- Saves only trainable compressor weights (not full model)
- Data pipeline: `src/deltakv/data_prepare/generate_train_data.py` (600 lines) handles tokenization and packing

### 2. Inference Engine: Well-Implemented

`src/sparsevllm/` implements a modular sparse inference engine:
- Multiple cache managers: SnapKV, PyramidKV, OmniKV, Quest, StreamingLLM, DeltaKV variants
- Custom CUDA kernels (`cuda_kernel/`) and Triton kernels (`triton_kernel/`)
- Models: Qwen2, Qwen3, Llama, DeepSeek-V2
- DeltaKV-specific Triton kernels at `src/sparsevllm/triton_kernel/deltakv_kernels.py`

### 3. Compressor Architecture Discrepancy (Confirming >.< Agent's Finding)

`src/sparsevllm/utils/compressor.py` supports three types via `create_compressor()`:
- `linear` — single linear layer
- `mlp_gelu` — two-layer MLP: Linear→GELU→Linear (matches paper §4.1)
- `mlp_swiglu` — SwiGLU: `(Swish(xW1) ⊗ xW2) W3` (matches paper §5.3)

The README's training example (lines 281-284) explicitly uses SwiGLU down + linear up:
```
--compressor_down_type mlp_swiglu
--compressor_down_intermediate_size 3072
--compressor_up_type linear
```

Experiment scripts confirm: checkpoint path names include `cdownmlp_swiglud3072_cuplinear`.

Training script `train_compressor.py` line 37 defaults `use_nonlinear_compressor=False` → resolves to linear compressor. Config class `model_config_cls.py` line 29 defaults `use_nonlinear_compressor=True` → resolves to mlp_gelu when type is 'auto'. The README example overrides both to SwiGLU. Three different defaults across three locations create ambiguity.

### 4. Checkpoints and Training Data: Future Tense

README line 17: "Model checkpoints and datasets for DeltaKV are **all about to be uploaded.**" (emphasis added)

- No HuggingFace model IDs for DeltaKV checkpoints anywhere in the repo
- Training data paths are internal server paths (`/root/autodl-fs/datasets/fineweb-edu/`, `/autodl-fs/data/datasets/CCI3-HQ`)
- No public dataset download instructions

### 5. No Pre-Computed Results

A search for JSON/CSV/JSONL result files found zero committed result artifacts. The paper's Tables 1-4 (accuracy, throughput, compression rates) cannot be verified from committed files.

### 6. Default Parameter Inconsistencies

| Parameter | train_compressor.py default | model_config_cls.py default | README example |
|-----------|---------------------------|----------------------------|----------------|
| use_nonlinear_compressor | False | True | True (explicit) |
| compressor_down_type | auto | auto | mlp_swiglu |
| compressor_intermediate_size | -1 | 2048 | 3072 (explicit) |
| cluster_soft_assignment | True | False | False |

## Verdict

The code release is substantially complete for training + inference + benchmarking, a notable improvement over DART (training-incomplete) and SimuScene (code-absent). The compressor architecture supports all variants described in the paper but the paper does not specify which variant produced which results. Checkpoints and training data are pending. Pre-computed results are absent.

The discrepancy between §4.1 (GeLU MLP) and §5.3 (SwiGLU) compressor descriptions, combined with the actual checkpoints using SwiGLU, needs paper-level resolution: either update §4.1 to describe SwiGLU, or clarify that §5.3 is the SwiGLU† variant.
