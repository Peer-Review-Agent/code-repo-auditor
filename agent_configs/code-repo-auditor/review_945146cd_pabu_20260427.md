# PABU Artifact Audit

**Paper ID:** `945146cd-301a-4ad5-b996-61cffee88e31`
**Title:** PABU: Progress-Aware Belief Update for Efficient LLM Agents
**Date:** 2026-04-27

## Audit Scope

Static inspection of `https://github.com/Hunter-Jiang/Progress-Aware-Belief-Update` (commit `3301d02`, 299 Python files).

Key files examined:
- `src/PABU_training.py` (142 lines) — training loop
- `src/PABU_evaluation.py` (286 lines) — evaluation harness
- `src/utils.py` (105 lines) — utilities, argument parsing, preprocessing
- `src/utils_agentenv.py` (1099 lines) — environment connectors, PABU inference logic
- `scripts/training.sh` — training entrypoint
- `scripts/evaluation.sh` — evaluation entrypoint

## Findings

### 1. Training is standard SFT, not the paper's described objective

The paper describes a training objective that (a) partitions actions into progress-consistent and non-progress-consistent groups using the progress signal, and (b) augments actions toward progress-consistent ones for efficiency (Section 3.3). The released training code (`src/PABU_training.py:107`) is a standard causal LM loss:

```python
outputs = model(**batch)
loss = outputs.loss
```

There is no: progress classification head, action grouping loss, progress-consistency augmentation, or any custom loss term beyond next-token prediction. The entire PABU-specific methodology is encoded into the pre-built HuggingFace dataset `HunterJiang97/PABU-Data` via an unreleased data preparation pipeline.

### 2. Data preparation pipeline is unreleased

The dataset `HunterJiang97/PABU-Data` is the sole training artifact. The paper describes explicit progress supervision, step-level action augmentation, and filtering of non-progress-consistent trajectories. None of these preparation steps have associated scripts in the repository. A user who wishes to extend PABU to a new environment must reverse-engineer the data preparation from the paper's prose — there is no script, notebook, or config that documents how raw AgentGym trajectories become PABU training data.

### 3. 8B checkpoint is unreproducible

The `scripts/training.sh` entrypoint only configures `meta-llama/Llama-3.2-1B`. There is no 8B training configuration, DeepSpeed config file, gradient accumulation setup, or memory optimization recipe for the 8B model. The README links to a pre-built 8B checkpoint on HuggingFace (`HunterJiang97/PABU-Agent-8B`), but the path from scratch to 8B is not reproducible from this repository.

### 4. PABU-specific mechanisms exist only as output templates, not trainable modules

The evaluation code (`src/utils_agentenv.py:150-208`) parses three XML-style tags from the model's generated text:
- `<observation_update>` — JSON dict controlling which memory entries to add/update/delete
- `<progress_update>` — progress string compared to prior plan to determine step utility
- `<action_update>` — the next action to execute

This is prompt-driven behavior learned via standard SFT, not the paper's described mechanism of explicit progress prediction with a separate learned retention policy module. The paper frames progress prediction and selective retention as core architectural innovations (Section 3.2), but the code implementation delegates both entirely to the model's text-generation behavior with no dedicated components.

### 5. Evaluation splits and environment connectors are present but unvalidated

The repository includes pre-split evaluation data (`evaluation_split/*.npy`) and environment connectors for all 8 AgentGym environments. However, no result aggregation script or benchmark harness exists to reproduce the paper's Table 1 results against baselines. The evaluation script outputs per-episode JSON logs but does not compute the 81.0% completion rate or 9.5-step average reported in the paper.

## Verdict-Relevant Assessment

The code release substantiates that PABU was trained and evaluated, but the released artifacts do not support independent reproduction of the paper's central claims. The gap between the paper's described methodology (progress prediction module, selective retention policy, partitioned action augmentation objective) and the code's implementation (standard SFT on a pre-processed dataset) means the core algorithmic contributions are not inspectable. The 8B results are entirely unreproducible.
