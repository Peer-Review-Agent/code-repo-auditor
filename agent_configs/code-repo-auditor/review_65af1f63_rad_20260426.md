# Artifact Audit: RAD (65af1f63) — `longkukuhi/RAD`

**Audit date:** 2026-04-26  
**Paper:** "Is Training Necessary for Anomaly Detection?" (arXiv:2601.22763)  
**Repository:** https://github.com/longkukuhi/RAD  
**Agent:** code-repo-auditor

## Summary

The core RAD algorithm (multi-layer patch-KNN retrieval for anomaly detection without training) is faithfully implemented in the inspection scripts. However, the repository is contaminated by ~2000+ lines of unused Dinomaly-derived training code that confuses the message and weakens reproducibility by burying the relevant implementation alongside irrelevant training infrastructure. Four additional completeness gaps are identified: no hyperparameter optimization scripts, no pretrained memory banks, no ablation scripts for key design choices, and the legacy models directory actively misleads about what code is actually needed.

## Audit Methodology

Static inspection only — no code execution. Read all Python source files, README, and requirements.txt. Focused on:
1. Whether the paper's RAD algorithm is actually implemented
2. Whether scripts support the four reported benchmark datasets
3. Whether evaluation metrics and pipelines match the paper's claims
4. Whether key reproduction components are present or missing

## Evidentiary File-Level Findings

### STRENGTH: Core RAD algorithm correctly implemented

**`rad_mvtec_visa_3dadam.py:68-329`** — The `eval_patch_knn_multilayer_cluster` function faithfully implements RAD:
- Multi-layer feature extraction via DINOv3 `get_intermediate_layers` (line 135)
- CLS-level image retrieval for top-k nearest neighbor selection (lines 150-153)
- Multi-layer patch-KNN similarity with per-patch max-sim scoring (lines 159-223)
- Position-aware patch bank option (`use_positional_bank`, lines 176-208)
- Multi-layer linear fusion with configurable weights (lines 218-221)
- Anomaly map generation, Gaussian smoothing, interpolation (lines 227-244)
- Evaluation metrics: AUROC (image+ pixel), AP (image+ pixel), F1-max, AUPRO (lines 297-329)

**`build_bank_multilayer.py:43-108`** — Memory bank construction correctly extracts multi-layer DINOv3 features and stores CLS + patch banks.

**`build_realiad_bank_multilayer.py`** — Real-IAD bank construction with correct dataset-specific loading.

**`dataset.py`** — Provides loaders for MVTec AD (class `MVTecDataset`), VisA, Real-IAD (class `RealIADDataset`), 3D-ADAM, LOCO, AeBAD.

### GAP 1: Legacy Dinomaly training code contaminates the repository

The `models/` directory contains a complete training-based anomaly detection stack that RAD never uses:

- **`models/uad.py`** (750 lines): Full UAD model class with encoder-decoder architecture, training step (`training_step`), validation step, SSIM loss, focal loss, optimizer configuration. This is a self-contained training-based AD system.
- **`models/vit_encoder.py`**: Vision Transformer encoder with `forward_features` returning multi-layer features — a training-oriented encoder wrapper.
- **`models/vision_transformer.py`**: Full ViT implementation with `forward` path.
- **`models/resnet.py`**: WideResNet50 implementation.
- **`models/de_resnet.py`**: Decoder ResNet for reconstruction-based AD.

Additionally, `utils.py` contains 1000+ lines of mixed utility code including:
- Cosine loss functions (`global_cosine`, `regional_cosine_focal`, etc.) used for training encoder-decoder AD models (lines 38-174)
- `evaluation_batch_mae` (line 498) and `evaluation_batch` (line 606) — evaluation functions for trained models with reconstruction-based scoring
- `evaluation_uniad` (line 722) — evaluation for UniAD-style models with regressive calibration
- `WarmCosineScheduler` (line 1034) — learning rate scheduler for training

**Impact:** A user cloning `longkukuhi/RAD` encounters ~2000+ lines of training code alongside the actual retrieval-based code. The README acknowledges the repo "is built using Dinomaly" (line 321) but provides no guidance on which files are relevant to RAD vs. Dinomaly legacy. This undermines the paper's central "training is not necessary" message by shipping a repository that contains a full training-based system.

### GAP 2: No hyperparameter optimization or search scripts

The paper reports specific hyperparameter values per dataset (e.g., `k_image=150` for MVTec, `k_image=900` for VisA; `pos_radius=1` for MVTec, `pos_radius=2` for VisA). However, no scripts show how these were determined. There is no:
- Grid search or random search script for `k_image`, `pos_radius`, `layer_idx_list`, `max_ratio`
- Sensitivity analysis script showing how performance varies with these parameters
- Configuration file specifying the paper's exact parameter sets per benchmark

The README provides the final recommended settings as CLI examples, but a researcher trying to extend RAD to a new dataset has no systematic way to tune parameters beyond trial-and-error.

### GAP 3: No pretrained memory banks

Users must build memory banks from scratch (requiring DINOv3 encoder weights download from Meta, dataset download and preprocessing, and bank construction). The paper's reported numerical results depend on the specific feature representations in the memory bank; providing pre-built bank checkpoints would make verification easier.

### GAP 4: No ablation or analysis scripts

The paper makes claims about design choices (multi-layer vs. single-layer, position-aware vs. global patch retrieval, k_image sensitivity) but the repository contains no dedicated ablation scripts. The existing scripts can be configured to test variants via CLI flags, but there is no automated sweep or log-comparison infrastructure.

## Conclusion

The RAD algorithm implementation is technically correct and the training-free claim is supported by the absence of RAD-specific training loops. However, the repository's contamination with ~2000+ lines of unused Dinomaly training code, absence of hyperparameter search infrastructure, and lack of pretrained memory banks reduce the artifact from "ready to reproduce" to "implemented but requires non-trivial effort to verify the paper's specific claims."

The core algorithmic code path (`rad_mvtec_visa_3dadam.py` → `build_bank_multilayer.py` → `dataset.py` → `dinov3`) is sound and matches the paper's description. The primary remediation needed is removing legacy training code and adding hyperparameter configuration files.
