# Code Artifact Audit: SPA (60121fd8)

## Paper
SPA: A Simple but Tough-to-Beat Baseline for Knowledge Injection
arXiv:2603.22213

## Repository
https://github.com/Tangkexian/SPA (17 files: 12 Python, 5 shell scripts)

## Audit Summary
The SPA repository implements the data generation and continued pretraining pipeline. The 7 prompt templates, shell scripts, and training loop are all present and inspectable. However, the repository contains **zero evaluation entrypoints** — no script, tool, or pipeline exists to reproduce the paper's benchmark results (SQuAD, QuALITY, MultiHop-RAG). The paper's central empirical claims cannot be independently verified from the released artifacts.

## Detailed Files Inspected

### Pipeline files (present and functional):
- `src/generate_data.py` (297 lines): vLLM/API-based data generation using prompt templates, resumable, well-parameterized
- `src/train/CPT_tokenize.py` (131 lines): Tokenization with chunking, produces HF Arrow datasets
- `src/train/CPT_train.py` (154 lines): Continued pretraining via Hugging Face Trainer with DeepSpeed support
- `src/utils_tools/prompt_utils.py` (289 lines): All 7 prompt templates in 4 format variants (Instruct, OSS, Base, API)
- Shell scripts: `make_squad_data.sh`, `make_quality_data.sh`, `make_mhrag_data.sh`, `tokenize.sh`, `train.sh`

### What is missing:

1. **Zero evaluation entrypoints.** No `evaluate.sh`, `eval.py`, or any script that runs benchmark evaluation. The grep for "evaluate", "eval", "SQuAD", "QuALITY", and "MultiHop" across all 12 Python files and 5 shell scripts returns no evaluation entrypoint calls. The paper evaluates on 3 benchmarks; the repo implements evaluation for 0.

2. **SQuAD evaluation utilities exist but are orphaned.** `src/utils_tools/utils.py` contains `format_answer_prompts()`, `grade_with_gpt4()`, `extract_final_answer()`, `parse_yes_no()`, and SQuAD answer/grading templates — all the building blocks for SQuAD evaluation. However, these functions are never called by any script in the repository. There is no entrypoint that loads a trained model, runs it on SQuAD questions, and computes accuracy.

3. **QuALITY and MultiHop-RAG evaluation code entirely absent.** The `src/tasks/quality.py` file is a data loader (QuALITY document/Question classes), not an evaluator. No evaluation utilities exist for MultiHop-RAG. A researcher must write evaluation code from scratch to reproduce results on two of the three benchmarks.

4. **QuALITY and MultiHop-RAG source data not included.** The `data/` directory contains only `squad_val_200.json` (200 SQuAD articles). The QuALITY dataset (`data/QuALITY`) and MultiHop-RAG dataset referenced in `make_quality_data.sh` and `make_mhrag_data.sh` must be obtained externally.

## Impact on Paper Claims

The paper's central empirical results (Tables 1-3, Figures 1-2, scaling curves) all depend on benchmark evaluation. Without evaluation scripts:
- The claimed accuracy improvements over baselines cannot be independently verified
- The scaling behavior (Figure 1: "SPA consistently improves with scale") cannot be reproduced
- The comparison against RL-based augmentation and multi-stage prompting baselines cannot be checked
- Even the paper's simplest result (SPA on SQuAD) cannot be replicated end-to-end

The release supports **method reproduction** (generate data + train) but not **result reproduction** (evaluate and compare). The missing piece is the evaluation pipeline, which is the bridge between the released model and the paper's empirical claims.

## Verdict
The repo is a genuine implementation of the SPA method, but the absence of evaluation entrypoints means the paper's headline claims — "SPA outperforms several strong and more complex baselines" — cannot be independently verified from the released artifacts.
