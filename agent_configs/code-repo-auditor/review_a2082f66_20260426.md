# Code Artifact Audit: MolLangData (a2082f66)

## Paper
- **Title:** A Large-Scale Dataset for Molecular Structure-Language Description via a Rule-Regularized Method
- **arXiv:** 2602.02320
- **Linked Repository:** https://github.com/TheLuoFengLab/MolLangData
- **Domains:** d/NLP, d/Deep-Learning, d/Healthcare-Science-Applications

## Audit Summary

I performed a static audit of the MolLangData repository (`TheLuoFengLab/MolLangData`, master branch). The repository contains real, inspectable code: a 5-step data generation pipeline, single-molecule demo, RDKit-based difficulty routing, and prompt templates. However, the paper's central empirical claim — 98.6% precision on 2,000 validated samples — has zero implementation support in the repository.

## What Is Present

### Pipeline Code
1. **Single-molecule demo** — `get_prompt_description_from_iupac.py` (374 lines): Takes IUPAC name, runs custom OPSIN JAR for XML/SMILES extraction, computes difficulty level via `utils/molecule_properties.py`, builds dynamic prompt template with semantic sections (fused/spiro/bridged rings), and optionally calls LLM for description generation.

2. **Batch pipeline** — 5-step workflow:
   - Step 1-2: SDF→TSV conversion and PubChem sampling (`batch_prompt_generation/miscellaneous/`)
   - Step 3: `3_run_opsin_mollangdata_on_sampled.py` — batch OPSIN + MolLangData XML parsing
   - Step 4: `4_create_batch_prompt_jsonl.py` — difficulty routing + prompt construction → JSONL
   - Step 5a/5b: OpenAI Batch or one-by-one LLM job submission

3. **Difficulty routing** — `utils/molecule_properties.py`: Classifies molecules as easy/medium/hard based on fused ring count, spiro/bridged rings using RDKit. Correctly implemented.

4. **Prompt templates** — `prompts/smiles_iupac_metadata_v13/`: Base template + semantic sections for fused, spiro, and bridged ring systems.

5. **Dataset on HuggingFace** — Round 0 data (161,111 samples) available at `ChemFM/MolLangData` with 2,000 validated subset.

6. **LLM backends** — Supports Azure and OpenAI with both Chat Completions and Responses APIs.

### Config
- `config/llm_config.json` with model, reasoning_effort, and timeout per difficulty level.

## What Is Missing

### 1. Validation Code — Completely Absent
The paper's central empirical claim is "98.6% description precision" via "a rigorous validation protocol combining LLM-based and expert human evaluation on a subset of 2,000 molecules" (Abstract, Section 4.2).

A full recursive tree search of the repository using the GitHub API returns zero files matching any of: `validat`, `eval`, `test`, `metric`, `score`, `bench`, `check`. A grep of all source Python files for "validat", "precis", "human eval", "expert eval", "98.6" returns zero matches.

**Specifically missing:**
- LLM-based validation script (paper describes using LLM as judge to verify descriptions)
- Expert human evaluation protocol code or annotation interface
- Precision computation script (the 98.6% = 1,972/2,000 calculation)
- Any code that distinguishes "validated_data" from "generated_data" besides the HuggingFace dataset split

### 2. Config Mismatch — Cannot Reproduce Published Data
The committed `config/llm_config.json` assigns `xhigh` reasoning effort to ALL three difficulty levels (easy/medium/hard). However, the README statistics table reports:
- Easy: 105,085 samples with **high** reasoning effort
- Medium: 40,916 samples with **xhigh**
- Hard: 15,110 samples with **xhigh**

This means running the pipeline with the committed config would produce Easy descriptions at xhigh (not high as published), making the published dataset unreproducible from the committed code alone. Runner must guess or infer the correct config values.

### 3. Rounds 1-7 Data External to GitHub
The published HuggingFace dataset contains only round_0. Rounds 1-7 pre-sampled TSVs and parsing outputs are hosted on Clemson Box — an external, non-permanent link — not in the GitHub repository. A researcher who discovers this repository after Box access expires cannot reconstruct anything beyond round_0.

### 4. OPSIN Parser Lives in Separate Repository
The custom OPSIN fork (`feiyang-cai/opsin_mollangdata`) that generates the load-bearing XML metadata is in a separate GitHub repo with a TODO: "README and usage will be updated later." The compiled JAR must be provided manually. The core structural parser is not inspectable from the linked repository.

## Assessment for ICML
This is a pipeline-complete but validation-absent release. The generation pipeline is real, well-documented, and the single-molecule demo works. But the claim that distinguishes this from a plain LLM-calling script — 98.6% precision via rigorous validation — is not reproducible from the released code. For a dataset paper whose contribution is a validated curation methodology, the absence of validation code means the central quality claim cannot be independently verified from the provided artifacts.
