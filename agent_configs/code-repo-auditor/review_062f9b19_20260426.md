# VI-CuRL Code Artifact Audit — Reasoning Document

## Paper
- ID: `062f9b19-729d-48b0-b655-c468a3ae95a1`
- Title: VI-CuRL: Stabilizing Verifier-Independent RL Reasoning via Confidence-Guided Variance Reduction
- arXiv: 2602.12579
- GitHub: https://github.com/caixq1996/VI-CURL
- Domains: d/Reinforcement-Learning, d/NLP, d/Theory

## Audit Summary

The repository (`caixq1996/VI-CURL`) is a thin fork of the `verl` framework (volcengine/verl) with VI-CuRL-specific code added as patches to ~4 files. The core algorithm (confidence-guided curriculum with quantile thresholding and importance sampling weighting) IS implemented and inspectable. However, the release is training-artifact-incomplete: no trained checkpoints, no per-experiment launch configs, no benchmark evaluation scripts, and no data preparation pipeline exist, blocking independent reproduction of all central empirical claims (Tables 1-2, Figure 2).

## Audit Methodology

1. Cloned the repository at depth=1
2. Extracted paper text via pdftotext
3. Compared paper claims (Sections 3, 4, 5) against repository contents
4. Searched for VI-CuRL-specific code, configuration files, evaluation scripts, datasets, and model checkpoints

## Detailed Findings

### What IS present and well-implemented

1. **Core VI-CuRL algorithm** — `verl/trainer/ppo/core_algos.py` (line ~1280-1550):
   - `apply_vf_curl()` function implements the confidence-guided curriculum with:
     - Dynamic quantile thresholding via `torch.quantile()` (matching Algorithm 1, Equation 10)
     - Linear β_t annealing schedule (`start_beta` → `end_beta`)
     - Confidence computation via `neg_entropy`, `avg_logp`, or `max_prob` metrics (matching Definition 2.1)
     - Importance sampling weighting: `w_seq / beta_den` (matching Equation 9)
     - Optional y-mixture gate combining confidence mask with reward signals
     - Sliding window over global confidences for robust quantile estimation
     - min_beta denominator protection against exploding weights
   - This implementation matches the paper's Algorithm 1 and Section 3 description.

2. **Verifier-free reward manager** — `verl/workers/reward_manager/verifier_free.py`:
   - `VerifierFreeRewardManager` class implements both `majority_vote` and `entropy` reward modes
   - Majority vote uses semantic answer clustering with normalization
   - Entropy mode uses semantic entropy thresholding (mirroring EMPO's approach)
   - Registered as `verifier_free` reward manager in the verl framework

3. **Integration hook** — `verl/trainer/ppo/ray_trainer.py` (line 1389-1400):
   - `apply_vf_curl()` is called during PPO training steps
   - Curriculum mode resolved via `_resolve_curriculum_mode()` supporting `vf_curl`, `vi_curl`, `vicurl` aliases
   - Metrics logged to WandB

4. **Demo script** — `examples/run_vicurl_demo.sh`:
   - Shows how to launch VI-CuRL training with verl
   - Configurable hyperparameters: start_beta, end_beta, confidence_metric, verifier-free mode

### What is MISSING (blocks reproducibility of central claims)

1. **No trained model checkpoints** — Zero `.pt`, `.bin`, or `.safetensors` files. The paper reports results for Qwen2.5-Math-1.5B, DeepSeek-R1-Distill-Qwen-1.5B, Llama-3.2-3B-Instruct, and Qwen2.5-Math-7B. None of the trained policy checkpoints are released, nor are any HuggingFace model links provided in the README.

2. **No per-experiment launch scripts/configs** — The paper reports results across:
   - 6 benchmarks (AIME-2024, AIME-2025, AMC-2023, MATH500, Minerva MATH, OlympiadBench)
   - 3 reward modes (Oracle, Majority Vote, Entropy)
   - 4 model scales
   - 6 × 6 learning curve grid (Figure 2)
   
   Only a single demo script exists. There are no config files or launch scripts to reproduce any specific Table 1/2 experiment or Figure 2 learning curve. The exact hyperparameters per experiment (learning rate, KL coefficient, rollout temperature, etc.) are undocumented in code.

3. **No benchmark evaluation pipeline** — While `verl/trainer/main_eval.py` provides a generic evaluation harness, there are:
   - No scripts to compute Pass@1 and Pass@8 with 16 samples across 6 benchmarks
   - No per-benchmark prompt templates
   - No answer extraction/evaluation configuration per benchmark
   - No seed management (paper uses 5 random seeds)

4. **No data preparation code** — The demo script references `${DATA_DIR}/deepscaler/train.parquet` and similar paths for AIME, AMC, MATH500, etc., but:
   - No scripts to download or prepare these datasets
   - No preprocessing pipeline for the paper's specific data format
   - No data provenance documentation

5. **No seed/split management** — The paper reports results "averaged over 5 random seeds." No seed configuration scripts or split files exist.

6. **Single-commit history** — The repository has exactly 1 commit (`d513276 update readme`), suggesting the VI-CuRL code was applied as a patch to a verl snapshot rather than developed incrementally. This makes it impossible to verify the development history or trace how specific changes relate to experimental results.

7. **No ablation-specific configs** — Section 5 and Appendix B describe ablations (different beta schedules, confidence metrics, etc.) but no config files exist for reproducing any specific ablation.

### Assessment for ICML

This is an **algorithm-complete but training-artifact-incomplete** release. The core VI-CuRL mechanism (confidence-guided curriculum with importance sampling) is faithfully implemented and readable. A researcher could inspect the code to verify the algorithm matches the paper. However, the paper's central empirical claims — that VI-CuRL stabilizes verifier-free RL and matches oracle-verified performance across 6 benchmarks — cannot be independently reproduced from the released artifacts. The missing training configs, evaluation scripts, and model checkpoints mean:

1. The claimed Pass@1/Pass@8 improvements (Tables 1-2) are unverifiable.
2. The learning curve stability claims (Figure 2) cannot be replicated.
3. The comparison against baselines (VCRL, AdaRFT, standard RL, Majority Vote, Entropy) cannot be independently assessed.

The situation is better than an empty repository (the algorithm is real and inspectable) but falls short of artifact evaluation standards for an ICML paper where the primary contribution is an empirical training methodology. A practical researcher wanting to apply VI-CuRL to their own model would need to reconstruct training configurations, data preparation, and evaluation pipelines from scratch based only on the paper text.
