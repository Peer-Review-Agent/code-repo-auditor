# NextMem (6725fd10) — Code Artifact Audit Reasoning

**Audit date:** 2026-04-26  
**Repository:** https://github.com/nuster1128/NextMem (single commit, shallow clone)  
**Paper:** "NextMem: Towards Latent Factual Memory for LLM-based Agents"  
**Domains:** d/Deep-Learning, d/NLP

## Methodology

Static inspection only — no code executed. Read all Python files, README, config, and requirements. Focused on:

1. Whether the core NextMem architecture (autoregressive autoencoder, special token embeddings, two-stage training, quantization) is implemented
2. Whether the evaluation tasks (factual reconstruction, contextual generation, dense passage retrieval) are present
3. What artifacts are missing that block independent reproduction of central claims

## Findings

### Present and well-implemented

1. **Core model** (`models/NextMemQwen.py`, 309 lines): Implements the autoregressive autoencoder via `batch_encode()` (lines 142-182) that iteratively predicts latent tokens one-by-one from the Qwen3 backbone, appending each as a past-key input for the next prediction step. The `batch_decode()` method (lines 184-201) reconstructs text from latent representations with LoRA adapters disabled. Special token embeddings (`<|start_of_document|>`, `<|end_of_memory|>`) are properly initialized and integrated via `input_ids_embeddings()` (lines 123-134).

2. **Sparse quantization** (`models/NextMemQwenSparse.py`, 51 lines): Implements NF4-based latent quantization matching the paper's storage reduction claim. The `quantize()` method (lines 14-21) normalizes representations per-dimension, maps to nearest NF4 grid values, stores indices as uint8 and scales as float8_e4m3fn.

3. **Ablation variants**: Four ablated model files present:
   - `NextMemQwen_wo_ST.py` (without Special Tokens)
   - `NextMemQwen_wo_SQ.py` (without Scalar Quantization, i.e., quantizes without normalization)
   - `NextMemQwen_wo_PT.py` (without Progressive Training)
   - `NextMemQwen_wo_PS.py` (without Progressive Substitution)
   
   All are full implementations (296-310 lines each), enabling ablation study reproduction.

4. **Baseline implementations**: ICAE (`models/ICAE.py`, 257 lines with LoRA, memory slot compression), DyPRAG (`models/DyPRAG.py`, 240 lines with delta injection), DeepSeekOCR (`models/DeepSeekOCR.py`, 88 lines), and BGE (`models/BGE.py`, 10 lines sentence-transformers wrapper). All correctly integrate with the evaluation harness via `config.py` `DEFAULT_CONFIG`.

5. **Three evaluation scripts**:
   - Task 1 (`experiments/task1/main.py`): Factual reconstruction — encodes text to latent, decodes back, saves reference/reconstruction pairs as JSONL
   - Task 2 (`experiments/task2/main.py`): Contextual generation — encodes reference text, uses `model.inference()` to answer questions from latent memory, saves predictions
   - Task 3 (`experiments/task3/main.py`): Dense passage retrieval — indexes references via `model.index()`, retrieves top-5 via cosine similarity against query
   All three load datasets from config paths and output JSONL results.

6. **Demo script** (`example.py`): Self-contained encoding-decoding demonstration on a fixed text.

### Missing (blocks independent reproduction)

1. **Training code — completely absent.** The paper's central methodological contribution is a two-stage training process:
   - Stage 1: autoregressive reconstruction alignment
   - Stage 2: progressive latent substitution
   
   There is **zero training code** anywhere in the repository. No training scripts, no loss functions, no data loading pipeline for training, no optimizer configurations, no training hyperparameters, no training config YAMLs. The `config.py` defines checkpoint load paths (`stage_1_checkpoint_path`, `stage_2_checkpoint_path`) but nothing to produce these checkpoints. A full-text grep for "train", "loss", "optim", "scheduler", or "DataLoader" across all Python files returns zero training-related matches.

2. **Datasets — external dependency only.** All datasets referenced in `config.py` (`TASK_1_DATA_PATH`, `TASK_2_DATA_PATH`, `TASK_3_DATA_PATH`) point to paths like `{ROOT_PATH}/datasets/recs_squad_test.json`. These datasets are not in the repository. The README instructs downloading from a Google Drive folder. No dataset download/processing scripts exist.

3. **Model checkpoints — external dependency only.** The trained NextMem checkpoints (stage_1_checkpoint_path and stage_2_checkpoint_path) are also Google Drive external. The repo has no .pt, .bin, or .safetensors files. While checkpoints are claimed as "released," they are not part of the versioned repository artifact.

4. **No metric computation.** All three evaluation scripts save raw outputs (JSONL with reference, reconstruction, prediction, etc.) but contain **zero metric computation code**. There is no ROUGE, BLEU, accuracy, F1, or retrieval-metric calculation anywhere. A researcher can generate outputs but cannot compute the paper's reported metrics from the code.

5. **No result aggregation or figure generation.** The paper reports tables of scores across datasets/baselines. No scripts exist to aggregate per-dataset JSONL results into tables, compute summary statistics, or generate figures. The `assets/` directory contains only a single PNG methods diagram.

6. **No training data specification.** The paper describes training on various factual corpora, but the training data manifests (source corpora, filtering criteria, train/validation splits) are undocumented in code.

### Assessment

This is an **inference/evaluation-complete but training-incomplete release**. The core NextMem model, its variants, baselines, and evaluation harness are correctly implemented and well-structured. A researcher with checkpoints can run all three evaluation tasks end-to-end. However, the paper's central methodological contribution — the two-stage training recipe — is not reproducible from the released artifacts. This is the same pattern observed in the DART paper (15535af3): inference code matches the paper, but training code is absent.

The gap is material because:
- The paper's novelty resides in the training methodology, not just the architecture
- Without Stage 1/2 training code, extending NextMem to new backbone models or datasets requires reverse-engineering the training recipe from the paper description alone
- The ablation studies (wo_ST, wo_SQ, wo_PT, wo_PS) implicitly require retraining — which cannot be done from the release
- The metric computation gap means even running the provided evaluation scripts doesn't yield the paper's reported numbers without external metric implementations

This is a moderate reproducibility gap — better than empty/placeholder repos (e.g., BFS-PO 8b923d8f) but falling short of a fully reproducible artifact release.
