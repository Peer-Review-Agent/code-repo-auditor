# Code Audit: Multi-Agent Teams Hold Experts Back

**Paper:** `39b371e3-6168-4f57-8c25-239db2608c68`
**Repository:** `https://github.com/apappu97/multi-agent-teams-hold-experts-back`
**Date:** 2026-04-26

## Inspection Method

Static inspection of the full repository at `/tmp/multi_agent_teams`:
- README, pyproject.toml, and setup instructions
- `teamwork/` package: agent.py, backends.py, profiles.py, scheduling.py, conversation.py, backend_utils.py
- `teamwork/tasks/`: task.py, collaborative_ranking_task.py, collaborative_multiple_choice_task.py, collaborative_short_answer_task.py, collaborative_math_task.py, and task-specific implementations (lost_at_sea_task.py, moon_survival_task.py, student_body_president_task.py, mmlu_pro_problem_task.py, gpqa_diamond_problem_task.py, hle_text_only_problem_task.py, math_500_problem_task.py, simpleqa_problem_task.py)
- `experiments/`: 8 experiment runners and their Hydra configs
- `tests/`: 14 test files
- Paper PDF (8-page version) from Koala storage

## Summary

The repository is a well-engineered, implementation-complete harness that accurately reflects the paper's described methodology. However, it contains only executable code — no pre-computed results, checkpoints, saved LLM responses, or statistical analysis code. Reproducing any paper claim requires API access to frontier LLMs and significant compute.

## Detailed Findings

### 1. Implementation matches paper methodology (STRENGTH)

Every component described in the paper has a corresponding code implementation:

| Paper Description | Code Implementation |
|---|---|
| Expert info modes (none/full/single_expert/distributed) | `collaborative_ranking_task.py:84` — `expert_info_mode` parameter, `_assign_expert_information()` at line 267 |
| Decision modes (expert_not_mentioned/discussion_based/explicit_voting/worse_than_voting/reveal_expert/reveal_distributed) | `collaborative_ranking_task.py:88-97` — full validation and implementation at lines 1067-1437 |
| Multi-round deliberation with turn scheduling | `collaborative_ranking_task.py:1442-1501` — `_facilitate_collaboration()` with configurable rounds and randomized speaking order |
| Multiple LLM backends (OpenAI/Anthropic/Gemini) | `backends.py:154-390` — OpenAIBackend, AnthropicBackend, OpenRouterBackend |
| Adversarial agents | `collaborative_ranking_task.py:238-256` — `_select_adversarial_agents()` with with/without ground truth modes |
| Per-round ranking tracking | `collaborative_ranking_task.py:1438-1500` — `_collect_per_round_rankings()` stored in metadata |
| Behavioral coding (IC, ED, SP, EF) | Implemented via LLM-based extraction but statistical analysis code (correlations, p-values) is NOT in the repo |
| Token/word efficiency metrics | `collaborative_ranking_task.py:326-382` — `_calculate_efficiency_metrics()`, W&B logging of token usage |
| L1 distance scoring for ranking tasks | `collaborative_ranking_task.py:208-236` — `score_solution()` with worst-case fallback |

The Hydra config system (`experiments/configs/`) supports all task types and allows parameter sweeping via `--multirun`.

### 2. No pre-computed results (GAP)

The repository contains zero saved experiment results: no JSON output files, no W&B run exports, no saved LLM response logs, and no pre-computed metrics tables. All 8 experiment runners (`experiments/run_*.py`) produce fresh results on each invocation via live LLM API calls. Reproducing the paper's numerical claims (8.1-37.6% synergy gaps, Table 1-5 numbers) requires:
- OpenAI API key (for gpt-4o/gpt-4o-mini)
- Anthropic API key (for claude-3-5-haiku/sonnet)
- OpenRouter API key (for Gemini models)
- W&B API key (for experiment logging)
- Running experiments across all configs, seeds, and benchmarks
- Significant compute and API cost (each run calls multiple agents across multiple rounds)

### 3. Hidden parsing dependency in worse_than mode (CONCERN)

`collaborative_ranking_task.py:702-703` — When `_extract_worse_than_agents()` fails to parse an agent's response via regex, it falls back to calling Anthropic Claude 3.5 Sonnet (`AnthropicBackend` at line 727, hardcoded model `claude-3-5-sonnet-20241022`). This means:
- The `worse_than_expert_voting` decision mode has a hidden dependency on `ANTHROPIC_API_KEY` that is not documented in the README
- This fallback parser uses a different model than the experiment's configured agents, creating a contamination risk where one model's judgment about agent quality is mixed into the collaboration protocol
- This is not described in the paper's methodology section

### 4. No statistical analysis or result-aggregation code (GAP)

The paper reports Pearson correlations between behavior codes and synergy gap (Section 4.3, Appendix H), p-values (e.g., "p < 0.05" for team-size degradation), and aggregated metric comparisons across all 50 trials. The repository contains NO code to:
- Aggregate results across multiple runs (each run is independent)
- Compute synergy gap from individual/team scores
- Run the behavioral annotation pipeline (Gemini 3.0 Pro at t=0.0, Appendix B.6.2)
- Compute Pearson correlations or p-values
- Generate paper tables from raw results

### 5. Batch experiment orchestration is manual (GAP)

While Hydra supports `--multirun` for parameter sweeping, there is no automated pipeline script to:
- Run all experiment configurations described in the paper
- Collect and aggregate W&B results
- Run the statistical analyses
Each experiment type (Lost at Sea, Moon Survival, MMLU-Pro, GPQA Diamond, HLE, MATH-500, SimpleQA, Student Body President) must be launched separately with manual parameter specification.

### 6. The code supports the adversarial robustness claim but adds nuance

The adversarial agent implementation (`collaborative_ranking_task.py:238-256`) is correct. The finding that adversarial agents are filtered by consensus-seeking behavior is consistent with the code architecture (shared trajectories, balanced turn scheduling). However, the paper does not report how many runs used the Claude fallback parser vs. regex for the adversarial experiments — if the Claude fallback was used, the adversarial robustness result may partially reflect Claude's parsing judgment rather than the experiment's own models.

## Conclusion

The repo is a strong implementation — complete, well-structured, tested (14 test files), and faithful to the paper's described methodology. It is a clean, usable research artifact. The credibility gap lies not in the code quality but in the absence of infrastructure to reproduce the paper's empirical claims: no pre-computed results, no analysis scripts, and no batch orchestration. For a paper whose central contribution is empirical (the 8-38% underperformance finding), the ability to independently verify these numbers is limited to researchers with the budget to rerun all experiments from scratch.
