# Code Artifact Audit: Simple Baselines are Competitive with Code Evolution

**Paper ID:** `0bb9fe86-b711-4b1f-bec5-035ec976f497`
**Title:** Simple Baselines are Competitive with Code Evolution
**Linked Artifact:** `https://github.com/codelion/openevolve`
**Audit Date:** 2026-04-26

## What the paper claims

The paper evaluates whether simple baselines ŌĆö IID random sampling (IID RS) and sequential conditioned sampling ŌĆö can match or beat elaborate code evolution pipelines (AlphaEvolve, ShinkaEvolve, OpenEvolve) across three domains: mathematical bound computation, agentic scaffold design, and MLE-bench-style ML engineering.

## What the linked repository contains

The linked GitHub repository (`https://github.com/codelion/openevolve`) at commit in the default branch as of 2026-04-26 contains:

- **The OpenEvolve code evolution framework**: Full MAP-Elites implementation (`openevolve/controller.py`, `openevolve/database.py`), LLM ensemble (`openevolve/llm/`), prompt sampler (`openevolve/prompt/sampler.py`), 50+ unit and integration tests, and 21 example problem directories.
- **Well-implemented code evolution machinery**: Configurable evolution parameters (population_size=1000, archive_size=100, num_islands=5), multi-model ensemble support, island migration, program database with diversity tracking, and extensive prompt templates.
- **Example problem domains**: AlphaEvolve mathematical problems (14 problems from AlphaEvolve Appendices A and B in `examples/alphaevolve_math_problems/`), MLX Metal kernel optimization (`examples/mlx_metal_kernel_opt/`), AlgoTune benchmark (`examples/algotune/`).

## What is ABSENT from the repository

### 1. Simple baseline implementations ŌĆö not present

A full-text case-insensitive search for the paper's baseline terminology across all `.py`, `.md`, `.yaml`, `.json`, and `.toml` files returns zero relevant matches:
- "IID" or "independently identically distributed" ŌĆö no matches
- "simple baseline" ŌĆö no matches (only unrelated "simple" usages)
- "random sample" ŌĆö no matches
- "conditioned sampling" ŌĆö no matches
- "majority vote" ŌĆö no matches

The IID random sampling baseline and sequential conditioned sampling baseline that produce the paper's central empirical results are not implemented anywhere in the repository.

### 2. Paper-specific evaluation pipeline ŌĆö not present

The paper's experimental methodology ŌĆö running IID RS and conditioned sampling against AlphaEvolve, ShinkaEvolve, and OpenEvolve across three domains ŌĆö has no manifestation in the repository:
- No evaluation scripts that compare baseline methods against code evolution methods
- No result aggregation or summary statistics scripts (for Tables 1-3 or equivalent)
- No config files specifying the baseline parameters used in the paper
- No benchmark harness for the three-domain comparison

### 3. No reference to the ICML paper anywhere in the repository

The README, pyproject.toml, and all source files make no mention of the paper "Simple Baselines are Competitive with Code Evolution" or its claims. The repository presents itself as a general-purpose code evolution tool, not as the artifact for a specific empirical comparison paper.

## Assessment

**Identity mismatch: the linked artifact is the tool BEING EVALUATED, not the experiment code that performed the evaluation.** The repository is a well-engineered code evolution framework (OpenEvolve), one of the methods the paper compares its simple baselines against. The paper's own simple baseline implementations and evaluation pipeline are not present.

This blocks independent verification of the paper's central empirical claim: that IID random sampling and sequential conditioned sampling match or beat sophisticated code evolution methods. Without the baseline implementations and comparison harness, a researcher cannot reproduce the paper's core experiment or verify its quantitative results.
