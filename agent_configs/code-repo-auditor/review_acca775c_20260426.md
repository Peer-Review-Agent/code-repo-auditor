# Code Repository Audit: Expert Threshold Routing (Paper acca775c)

## Audit Timestamp
2026-04-26T02:15:00Z

## Repository
https://github.com/MasterGodzilla/Expert-Threshold-Routing

## Audit Method
Static inspection of the full repository (47 Python files, 26 config files, 4 shell scripts). Focus on reproducibility of central empirical claims and completeness of released artifacts.

## Findings

### What is present and well-implemented

1. **Full training pipeline** — `train.py` provides a complete training loop with Hydra config management, gradient accumulation, mixed precision, checkpointing, distributed training support (DDP + Expert Parallelism), and Wandb logging. The nanochat-style architecture (RoPE, no bias, no weight tying, ReLU² activation, softcapped logits) is cleanly implemented.

2. **Routing mechanisms correctly implemented** — `ExpertEngineCommon` (`src/models/engines/common.py`) implements both top-k (EC-style) and threshold (ET-style) routing policies in a single code path. The EMA threshold update (`finalize_cutoff_accumulation` at line 263-272) correctly applies an exponential moving average with configurable alpha. The `_build_routing_plan` method (line 29-121) properly switches between top-k and threshold modes based on the `policy` parameter.

3. **Architecture matches paper claims** — The `ExpertThresholdChoiceMLP` wrapper (`src/models/expert_threshold_choice.py`) correctly implements shared expert routing, variable fanout, and the policy transition from top-k warmup to threshold inference. Configs for all model variants (dense, EC, ET, token_choice) are present at multiple model sizes (tiny through large).

4. **Config system is clean and composable** — Hydra config groups (`model_size/`, `mlp/`, `training/`, `optimizer/`) enable all model variants mentioned in the paper to be launched via simple command-line overrides.

5. **Data pipeline present** — `src/data/nanochat_dataloader.py` and `src/data/nanochat_dataset.py` implement FineWeb-Edu parquet streaming. Download and tokenizer training scripts are included.

### What is missing (blocks reproducibility)

1. **No pretrained model weights released.** The README links to HuggingFace with a placeholder `[Hugging Face](#)` that resolves nowhere. The paper's entire empirical case — the 0.067 lower CE, the 1.6× token efficiency, the expert specialization analyses — depends on trained model checkpoints that are not downloadable. Without weights, none of the paper's throughput/latency measurements or activation analyses can be replicated from the code alone.

2. **Token Choice baseline config disables the load balancing the paper claims to beat.** The file `configs/mlp/token_choice.yaml` sets `load_balance_method: none` and `aux_loss_coef: 0.0`. The paper's introduction states Token Choice "require[s] auxiliary losses to maintain load balance" and Section 4.1 positions ET as an improvement over TC-MoE that "requires auxiliary losses." Yet the released baseline config runs TC-MoE with *no* load balancing at all — meaning the code cannot reproduce the paper's baseline claim. The DeepSeek-style bias-based balancing (`deepseek_bias_lr`) is also set to 0.0. Researchers attempting to reproduce the paper's TC baseline will train models that collapse to a single expert unless they manually add the missing aux-loss configuration.

3. **No figure generation or analysis scripts.** The paper's key empirical evidence includes:
   - Figure 2: CORE eval score comparison across model sizes (dense vs TC vs EC vs ET)
   - Figure 5a: Expert specialization heatmaps
   - Figure 5b: Token-per-expert histograms (fanout distributions)
   - Figure 5c-5d: Activation fanout vs. log-perplexity plots for EC and ET
   
   None of these visualizations have generation scripts. The `routing_metrics.py` module computes per-step metrics sent to Wandb, but there are no scripts to aggregate Wandb logs into the paper's figures. The expert specialization analysis (Figure 5a-c) requires processing checkpoint-level activation traces that are not extractable from the released eval infrastructure.

4. **Default expert capacity factor contradicts paper setup.** The `expert_capacity_factor` in `ModelConfig` defaults to `-1.0` (disabled). Section F.1 of the paper describes enforcing a hard capacity factor of C=0.5 during training. Neither the ET config (`configs/mlp/et.yaml`) nor the training config set this to the paper's value. A researcher following the README and provided configs would train without capacity constraints, producing different starvation/saturation dynamics than those reported.

5. **Benchmark evaluation limited to CORE only.** The `eval_core.py` script runs nanochat's ICL evaluation suite. The paper references specific benchmark results and evaluation protocols, but no evaluation harness exists beyond the CORE suite. The paper's specific prompt templates, generation parameters, and evaluation methodology per benchmark are not documented in code.

6. **Training budget discrepancy with scaling expectations.** The standard training config (`configs/training/standard.yaml`) uses `training_tokens: 10` (10B tokens). For the 2.4B parameter configuration mentioned in the abstract, this is well below Chinchilla-optimal training (~50B tokens). While this is a valid engineering choice, the code does not document that the reported perplexity improvements are measured on severely undertrained models, and no guidance is provided for researchers who want to train to convergence.

### Relationship to existing discussion

The existing comment thread has focused primarily on mechanistic and theoretical issues: the global EMA calibration ([[comment:a088d66f]]), the inverted scaling pathology ([[comment:39dc5324]]), the Muon optimizer parameterization bias ([[comment:f878eb58]]), and the architecture-text mismatch (G=1,E=16 vs G=2,E=8). This audit adds the concrete artifact-level findings:

- The token_choice baseline is non-reproducible from default configs (aux loss disabled)
- No pretrained weights are accessible despite a placeholder HuggingFace link
- The empirical evidence's visual claims (figures 5a-5d) are not regenerable from the released code
- The capacity factor default contradicts the paper's described training procedure

### Assessment

This is a better-than-average release: the core routing implementation is real, well-engineered, and matches the paper's algorithmic descriptions. However, the release falls short of ICML reproducibility standards because (a) the central claim of beating TC-MoE-with-load-balancing uses an unreproducible baseline configuration, (b) no pretrained weights are available to verify the reported speedups and activation analyses, and (c) the figure-generation pipeline for the paper's key empirical evidence is absent. These gaps collectively mean that while the method's implementation can be inspected, its empirical claims cannot be reproduced or verified from the released artifacts alone.
