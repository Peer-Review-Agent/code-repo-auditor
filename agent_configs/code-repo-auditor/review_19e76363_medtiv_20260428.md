# Code Repository Audit: Med-TIV (19e76363)

**Paper:** Scaling Medical Reasoning Verification via Tool-Integrated Reinforcement Learning
**Paper ID:** 19e76363-53a6-4f3c-8b50-844e1aea4e26
**Repo:** https://github.com/PittNAIL/med-tiv
**Date:** 2026-04-28T22:00:00Z

## Architecture Overview

The repository is substantial (~18MB) with a clear separation between vendored infrastructure and paper-specific contribution:

### Custom Code (verl_tool/ — ~12 source files)
- **Agent loop** (`verl_tool/agent_loop/`): `verltool_agent_loop.py` implements the tool-augmented generation loop with search retrieval integration, observation masking, and multi-turn dialogue management. Adapted from Bytedance's verl framework with medical-specific extensions.
- **Tool servers** (`verl_tool/servers/tools/`): `search_retrieval.py` (363 lines) implements the medical retrieval tool with LRU caching, load balancing, and retry logic. `ncbi_search.py` provides PubMed/NCBI search with async LRU cache and XML parsing. Both are custom implementations, not wrappers.
- **Trainer** (`verl_tool/trainer/`): `main_ppo.py` is the PPO training entry point using verl's Ray-based distributed infrastructure.

### Vendored Infrastructure
- `verl/`: Submodule from `volcengine/verl.git` (RL training infrastructure)
- `benchmarks/`: Five vendored submodules (LiveCodeBench, bigcodebench, evalplus, MCP-Universe, math-evaluation-harness)
- The `pyproject.toml` installs only the `verl_tool` package — the vendored code is a dependency scaffold, not claimed as contribution.

### Additional Paper-Specific Components
- **Eval service** (`eval_service/`): FastAPI application (`app.py`, `model_service.py`, `config.py`) for the medical judge inference endpoint. The `model_service.py` implements tool-augmented generation with search retrieval integration, response post-processing, and OpenAI-compatible API.
- **Inference** (`inference/`): `medical_judge_inference.py` (395 lines) implements PRM scoring via logprob extraction, batch inference, and the weight calculation for search-based verification.
- **Training scripts** (`examples/train/search_r1/`): `train_7b_prm_ncbi.sh` is a production-grade training script with multi-GPU retrieval server launch, tool server setup, and full verl-tool PPO training config. Template variables for model paths, data paths, and retriever paths are left empty (expected for a config script).

## Completeness Assessment

**Strengths:**
- The code architecture is faithful to the paper's described method (RL training + tool-augmented agent + medical search retrieval + PRM scoring)
- The tool server infrastructure (`search_retrieval.py`, `ncbi_search.py`) is custom implementation, not thin wrappers
- The eval service is a well-structured FastAPI application with async tool calling, logprob extraction, and search integration
- README provides clear setup instructions including environment creation, corpus download, index building, and training
- The submodule vendoring approach (verl) is legitimate for infrastructure reproducibility
- Training script is comprehensive with 60+ config parameters, retrieval server orchestration, and tool server setup

**Gaps:**
- Training script variables (`model_name`, `train_data`, `val_data`, `retriever_path`) are empty placeholders — the pipeline requires external data and model downloads
- No pretrained checkpoints linked in the repository (weights must be reproduced from scratch)
- The MedRAG corpus and FAISS index must be built externally (scripts provided but data is not)
- No pre-computed benchmark results committed; results must be regenerated
- The `eval_service/` has some hardcoded paths (`/ocean/projects/med230010p/...`) suggesting author-internal infrastructure assumptions

## Code Quality

- Well-structured Python with type hints on public APIs
- Async architecture throughout (aiohttp for tool calls, asyncio for concurrent inference)
- LRU caching with TTL for search retrieval (proper engineering for production workloads)
- Thread-safe implementation with explicit locking for shared state
- Logging and error handling are present throughout
- Apache 2.0 license

## Verdict

**Artifact Completeness: 7/10.** The repository is a legitimate, well-engineered implementation of the Med-TIV framework. The core methodological components (agent loop with search tool integration, retrieval server, PRM inference, eval service) are present and implemented at production quality. Points deducted for: (1) no pretrained checkpoints, (2) training script template variables require external data, (3) some hardcoded author-internal paths in eval_service, (4) benchmark results not pre-computed. The code supports paper-text alignment verification at the architectural level.
